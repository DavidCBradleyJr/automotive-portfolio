# Phase 5: Cloudinary Storage & Build Pipeline - Research

**Researched:** 2026-03-16
**Domain:** Cloudinary CDN image storage, Node.js Admin/Search API, build-time data generation
**Confidence:** HIGH

## Summary

This phase migrates 29 gallery images from the git repository to Cloudinary CDN and replaces the Sharp-based `process-images.js` with a new `build-gallery-data.js` script that fetches metadata from Cloudinary's Search API and generates the same `gallery-images.js` format. The gallery component requires zero changes -- it continues to import the static data file.

The Cloudinary Node.js SDK (`cloudinary` npm package) provides Admin API and Search API methods that return image dimensions, public IDs, folder paths, and contextual metadata (caption/alt). LQIP base64 strings are generated by fetching a tiny Cloudinary-transformed thumbnail URL (w_16, e_blur, q_auto, f_webp) and converting the response buffer to base64. All Cloudinary SDK usage is build-time only -- the public site JS bundle is unaffected.

**Primary recommendation:** Use contextual metadata (not structured metadata) for caption/alt storage -- it is simpler key-value pairs set per-asset during upload, requires no global field definitions, and is returned via `with_field('context')` in search results.

| Library | Version | Purpose | Why Standard |
|---------|---------|---------|--------------|

### Supporting
| Library | Version | Purpose | When to Use |
|---------|---------|---------|-------------|

### Alternatives Considered
| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| Contextual metadata | Structured metadata | Structured requires global field definitions, overkill for caption/alt key-value pairs |
| Search API | Admin API resources_by_asset_folder | Search API is more flexible, returns width/height by default, supports with_field |
| node fetch (built-in) | axios/node-fetch | No external dependency needed -- Node 18+ has global fetch for LQIP downloads |

**Installation:**
```bash
npm install --save-dev cloudinary
```

Note: `dotenv` is optional. Netlify injects env vars during build. For local dev, a `.env` file with `CLOUDINARY_CLOUD_NAME`, `CLOUDINARY_API_KEY`, `CLOUDINARY_API_SECRET` is sufficient using Node's `--env-file` flag (Node 20+) or dotenv.

### Pattern 1: Cloudinary SDK Configuration (ESM)
**What:** Initialize Cloudinary SDK using environment variables
**When to use:** At the top of any build script that needs Cloudinary access
**Example:**
```javascript
// Source: Cloudinary Node.js SDK docs + npm package
import { v2 as cloudinary } from 'cloudinary';

cloudinary.config({
  cloud_name: process.env.CLOUDINARY_CLOUD_NAME,
  api_key: process.env.CLOUDINARY_API_KEY,
  api_secret: process.env.CLOUDINARY_API_SECRET,
});
```

### Pattern 2: Search API for Gallery Data Fetch
**What:** Fetch all gallery images with metadata, dimensions, and folder info
**When to use:** In the build script to gather all image data
**Example:**
```javascript
// Source: Cloudinary Search API docs
const result = await cloudinary.search
  .expression('folder:gallery/*')
  .with_field('context')
  .max_results(100)
  .sort_by('public_id', 'asc')
  .execute();

// result.resources contains array of:
// { public_id, asset_id, folder, filename, format, width, height,
//   secure_url, context: { custom: { caption: '...', alt: '...' } } }
```

### Pattern 3: Cloudinary CDN URL Construction
**What:** Build optimized delivery URLs with transforms
**When to use:** When generating src field for gallery data entries
**Example:**
```javascript
// URL format:
// https://res.cloudinary.com/{cloud}/image/upload/{transforms}/{public_id}.{ext}
//
// With transforms:
// https://res.cloudinary.com/{cloud}/image/upload/f_auto,q_auto,w_2000,c_limit/gallery/jdm/jdm-street-racer-01

// Using SDK helper:
const url = cloudinary.url(resource.public_id, {
  fetch_format: 'auto',
  quality: 'auto',
  width: 2000,
  crop: 'limit',
  secure: true,
});
```

### Pattern 4: LQIP Base64 Generation from Cloudinary Transform
**What:** Fetch a tiny blurred thumbnail and convert to base64 data URL
**When to use:** During build for each gallery image
**Example:**
```javascript
// Build a tiny transform URL
const lqipUrl = cloudinary.url(resource.public_id, {
  width: 16,
  effect: 'blur:1000',
  quality: 'auto',
  fetch_format: 'webp',
  secure: true,
});

// Fetch and convert to base64
const response = await fetch(lqipUrl);
const buffer = Buffer.from(await response.arrayBuffer());
const lqip = `data:image/webp;base64,${buffer.toString('base64')}`;
```

### Pattern 5: Migration Upload Script
**What:** One-time script to upload existing 29 images to Cloudinary with metadata
**When to use:** During initial migration (run once by developer)
**Example:**
```javascript
// Upload with folder path, public_id, and context metadata
const result = await cloudinary.uploader.upload(
  `public/images/gallery/${filename}`,

    public_id: id,              // e.g. 'jdm-street-racer-01'

    overwrite: true,

```

### Pattern 6: Netlify Build Integration
**What:** Run build-gallery-data.js before Vite build
**When to use:** In package.json build script
**Example:**
```json
{
  "scripts": {
    "prebuild": "node scripts/build-gallery-data.js",

```

### Anti-Patterns to Avoid
- **Importing cloudinary SDK in frontend code:** The SDK is 500KB+, must be devDependency only. Gallery component imports static data file, never the SDK.
- **Storing API keys in code or .env committed to git:** Use Netlify env vars. Add `.env` to `.gitignore`.
- **Using Admin API `resources()` instead of Search API:** The Search API is more powerful, returns dimensions by default, and supports `with_field` for metadata.
- **Generating transforms at runtime:** All URL construction happens at build time. The gallery data file contains final CDN URLs.
- **Fetching LQIP sequentially:** With 29 images, sequential fetch takes ~15-30s. Use Promise.all with concurrency limit (batch of 5-10).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Image format negotiation | Custom format detection | Cloudinary `f_auto` | Handles WebP/AVIF/JPEG based on browser Accept header |
| Image resizing | Server-side Sharp pipeline | Cloudinary `w_2000,c_limit` | On-demand, cached at CDN edge |
| Quality optimization | Manual quality tweaking | Cloudinary `q_auto` | ML-based perceptual quality optimization |
| CDN delivery | Self-hosted or S3+CloudFront | Cloudinary CDN | Built-in, no configuration needed |
| LQIP generation | Local Sharp resize | Cloudinary URL transform + fetch | No local image processing needed |

**Key insight:** The entire Sharp-based pipeline (`process-images.js`) is replaced by Cloudinary URL transforms. No local image processing is needed at build time -- just API calls and URL construction.

## Common Pitfalls

### Pitfall 1: BTS and Social Wall Image Paths Break After Migration
**What goes wrong:** `index.html` has 13 hardcoded `<img>` tags referencing `/images/gallery/*.webp` for BTS (4 images) and Social Wall (9 images) sections. When `public/images/gallery/` is removed from git, these images 404.
**Why it happens:** CONTEXT.md says "BTS images stay hardcoded" and "Social wall images stay hardcoded" -- meaning they should NOT be managed by the gallery pipeline but they still need to be served.
**How to avoid:** Either: (a) copy the 13 referenced images to a separate `/images/bts/` and `/images/social/` folder and update the HTML paths, OR (b) update the HTML `src` attributes to Cloudinary CDN URLs directly. Option (b) is cleaner since the images are already on Cloudinary after migration.
**Warning signs:** 404s on BTS and social wall images after deploy.

### Pitfall 2: Search API Pagination (Max 500 Results)
**What goes wrong:** If gallery grows beyond 500 images, search results are truncated.
**Why it happens:** Search API returns max 500 results per request (not 100 as the default `max_results` suggests -- the default is 50, max is 500).
**How to avoid:** Set `max_results(500)` and implement `next_cursor` pagination. For 29 images this is not an issue now, but future-proof the script.
**Warning signs:** Missing images in generated data file.

### Pitfall 3: Cloudinary Context Metadata Pipe Character
**What goes wrong:** If caption or alt text contains `|` or `=` characters, the context string parsing breaks.
**Why it happens:** Context is stored as `key1=value1|key2=value2` format -- pipe and equals are delimiters.
**How to avoid:** Escape special characters or use the structured API call format: `context: { caption: '...', alt: '...' }` as an object (supported in Node SDK).
**Warning signs:** Truncated or missing caption/alt in generated data.

### Pitfall 4: Missing Environment Variables in Build
**What goes wrong:** Build script crashes because CLOUDINARY_CLOUD_NAME/API_KEY/API_SECRET are not set.
**Why it happens:** Env vars not configured in Netlify, or local dev missing `.env` file.
**How to avoid:** Fail fast with a clear error message at script start. Check all three env vars before making API calls.
**Warning signs:** Build failures on Netlify deploy.

### Pitfall 5: Hero Image LCP Regression
**What goes wrong:** Hero image currently at `/images/hero-placeholder.webp` with `fetchpriority="high"`. If moved to Cloudinary, the first load requires DNS resolution + TLS to `res.cloudinary.com`, adding latency to LCP.
**Why it happens:** Local images benefit from same-origin delivery, no extra DNS.
**How to avoid:** Keep hero image local (it is NOT a gallery image -- it is a separate file at `/images/hero-placeholder.webp`). It has `fetchpriority="high"` and is the LCP element. Moving it to Cloudinary adds unnecessary risk for minimal benefit.
**Warning signs:** LCP regression in Lighthouse after deploy.

### Pitfall 6: Folder Mode Mismatch (Fixed vs Dynamic)
**What goes wrong:** Cloudinary has two folder modes: "fixed" (legacy) and "dynamic". In fixed mode, `folder` in upload sets the path prefix of `public_id`. In dynamic mode, `folder` is separate from `public_id`.
**Why it happens:** New accounts default to dynamic folder mode since 2023.
**How to avoid:** Be aware of which mode the account uses. In dynamic mode, `public_id` does NOT include the folder path. The `asset_folder` field in search results gives the folder. Use the Search API `folder:gallery/*` expression which works in both modes.
**Warning signs:** public_id includes or excludes folder path unexpectedly.

### LQIP Parallelization with Concurrency Limit
```javascript
async function generateLqips(resources, concurrency = 5) {
  const results = [];
  for (let i = 0; i < resources.length; i += concurrency) {
    const batch = resources.slice(i, i + concurrency);
    const batchResults = await Promise.all(
      batch.map(async (resource) => {
        const lqipUrl = cloudinary.url(resource.public_id, {
          width: 16,
          effect: 'blur:1000',
          quality: 'auto',
          fetch_format: 'webp',
          secure: true,
        });
        const response = await fetch(lqipUrl);
        const buffer = Buffer.from(await response.arrayBuffer());
        return `data:image/webp;base64,${buffer.toString('base64')}`;
      })
    );
    results.push(...batchResults);
  }
  return results;
}
```

## Open Questions

1. **Dynamic vs Fixed Folder Mode**
   - What we know: New accounts (2023+) default to dynamic folder mode
   - What's unclear: Exact behavior difference for public_id construction in search results
   - Recommendation: Test during migration. In dynamic mode, use `asset_folder` field from search results to derive category. The Search API `folder:gallery/*` expression works in both modes.

2. **Cloudinary Context Object Form in Upload**
   - What we know: Context can be passed as string (`caption=Foo|alt=Bar`) or object (`{ caption: 'Foo', alt: 'Bar' }`)
   - What's unclear: Whether the object form is fully supported in all SDK versions
   - Recommendation: Use object form. If it fails, fall back to string form with escaped special characters.

3. **Build Caching Between Deploys**
   - What we know: Each Netlify build runs the script fresh, fetching all 29 images from Cloudinary
   - What's unclear: Whether Netlify build cache can persist generated gallery-images.js
   - Recommendation: Skip caching for now. 29 images + LQIP fetches should complete in under 30 seconds. Optimize only if build times become a problem.
